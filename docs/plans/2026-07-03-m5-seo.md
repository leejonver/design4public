# Milestone 5: SEO Optimization (structured data, photo detail pages, sitemap, manifest) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Complete spec §8's SEO work: (1) build the `/photos/[id]` photo detail page (currently an empty stub dir) so project photos are individually indexable and crawlable from the photo grid, (2) add JSON-LD structured data to every detail page (item → `Product`, project → `Article` + `ImageObject`, brand → `Brand`, photo → `ImageObject`, **all** detail pages → `BreadcrumbList`), (3) add photo URLs and real `lastModified` timestamps to the sitemap, (4) add `app/manifest.ts`, and (5) extend the E2E suite. `robots.ts` is already at the app root (done in M3) — **out of scope**.

**Architecture:** JSON-LD is generated by pure helpers in `lib/seo.ts` (which already exports `organizationSchema`, `websiteSchema`, `breadcrumbSchema`, `jsonLdGraph` and the unused-but-defined `breadcrumbSchema`). New helpers (`articleSchema`, `productSchema`, `brandSchema`, `imageObjectSchema`) follow the existing convention: each returns a bare node **without** `@context`, and the page wraps a list of them in `jsonLdGraph([...])` — one `<script type="application/ld+json">` per page (rendered via the existing `components/json-ld.tsx`). The site layout already emits an org+website graph on every page, so detail pages will carry **two** JSON-LD scripts (layout graph + page graph) — expected and correct.

The photo detail page reuses the existing data-layer patterns in `lib/api.ts`: a new `fetchPhotoById(id)` returns the already-defined `PhotoDetail` type (`PhotoFeedItem & { items: ItemSummary[] }`). It embeds the photo's published project (via `project_photos → projects`, published-only) and its tagged items (via `photo_items → items`, reusing `ITEM_SUMMARY_SELECT` + `normalizeItemSummary`). To satisfy "real `lastModified`", `updatedAt` is threaded onto the four summary/feed view-models from each table's existing `updated_at` column.

Photo grid tiles become real crawlable `<Link href="/photos/[id]">` anchors that **preventDefault → open the existing lightbox** on normal click (SEO-visible href; unchanged lightbox UX; cmd/middle-click opens the detail page).

**Revalidation:** the new `/photos/[id]` page reads the `photos` table, so its cached fetch carries the `sb:photos` tag (see `lib/supabase/public.ts` tagging + `lib/revalidation.ts`). Photo mutations must now also purge `sb:photos` (photo detail bodies) **and** the sitemap (which now lists photos); project/item/category mutations must purge `sb:photos` too because photo detail pages render the project title/publish-gate and the tagged-items block. Per the M2/M4 lesson, cross-entity dynamic purges use `revalidateTag('sb:<table>')`, never a dynamic-pattern `revalidatePath`.

**Tech Stack:** Next.js 14.2.32 (App Router), `@supabase/supabase-js` (site reads via anon client), `next/og` (existing OG image), Vitest 3, Playwright 1.55, Supabase CLI local stack (Docker). No new dependencies. **No SQL migrations** (`updated_at` columns already exist on `projects`/`items`/`brands`/`photos` — verified in `lib/database.types.ts`).

## Global Constraints

- **Production DB is untouchable — NO migration in M5.** Every column M5 reads (`updated_at`) already exists in the M3 baseline schema.
- **Branch:** all work on `unify/m5-seo`, branched from `unify/m4-data-model`. Frequent commits per task.
- **Korean locale throughout.** All user-visible strings Korean; JSON-LD `inLanguage: "ko-KR"`; OG `locale: "ko_KR"` (already set in `createPageMetadata`).
- **Public-photo definition is preserved and enforced.** A photo is public iff it belongs to a **published** project (mirrors `fetchPhotos`'s feed filter). The indexable set (`/photos/[id]` returning 200), the sitemap photo set, and the grid-linked set are therefore **identical**. Item-gallery-only photos (in no published project) get **no** detail page (→ `notFound()`), are **not** in the sitemap, and keep only their existing lightbox/`DetailHero` presentation on item pages. `fetchPhotoById` enforces `projects.status === 'published'` explicitly (not relying on RLS) for determinism.
- **No offers/price on `Product`.** These are public-procurement furniture reference pages with no e-commerce price — omit `offers`, `aggregateRating` (spec §8, lead scope item 3).
- **JSON-LD nodes carry no `@context`** — only `jsonLdGraph()` sets it once at the graph root (matches existing `organizationSchema`/`websiteSchema`).
- **Surgical.** Do not restyle detail pages; JSON-LD `<script>` tags are invisible. The only visual change is photo grid tiles becoming anchors (identical appearance via retained `className`) and the new photo detail page.
- **Revalidation is best-effort** (`revalidateEntity` already swallows errors) — never fail a mutation on a purge error.
- **Verification commands:** typecheck `npx tsc --noEmit` (expect 0 errors); unit `npx vitest run` (expect all green); e2e `npx playwright test` against the local stack (`supabase start` + seeded, `next dev` via `playwright.config.ts` webServer). No `typecheck` npm script exists — call `tsc` directly. Gate requires exclusive ownership of the running stack.

```ts
/** Strip cache-busters and drop empties — JSON-LD image URLs should be canonical. */
function schemaImages(images: Array<string | null | undefined>): string[] | undefined {
  const urls = images
    .map(stripCacheBuster)
    .filter((u): u is string => Boolean(u));
  return urls.length > 0 ? urls : undefined;
}

export function articleSchema({
  headline,
  description,
  images = [],
  datePublished,
  dateModified,
  path,
}: {
  headline: string;
  description: string | null;
  images?: Array<string | null | undefined>;
  datePublished?: string;
  dateModified?: string;
  path: string;
}) {
  return {
    "@type": "Article",
    headline,
    description: description ?? undefined,
    image: schemaImages(images),
    datePublished,
    dateModified,
    mainEntityOfPage: absoluteUrl(path),
    inLanguage: "ko-KR",
    author: { "@id": `${SITE_URL}/#organization` },
    publisher: { "@id": `${SITE_URL}/#organization` },
  };
}

export function productSchema({
  name,
  description,
  images = [],
  brand,
  path,
}: {
  name: string;
  description: string | null;
  images?: Array<string | null | undefined>;
  brand: string | null;
  path: string;
}) {
  // No `offers` / `aggregateRating`: public-procurement reference pages, no price.
  return {
    "@type": "Product",
    name,
    description: description ?? undefined,
    image: schemaImages(images),
    brand: brand ? { "@type": "Brand", name: brand } : undefined,
    url: absoluteUrl(path),
  };
}

export function brandSchema({
  name,
  nameEn,
  description,
  logo,
  website,
  path,
}: {
  name: string;
  nameEn: string | null;
  description: string | null;
  logo: string | null;
  website: string | null;
  path: string;
}) {
  return {
    "@type": "Brand",
    name,
    alternateName: nameEn ?? undefined,
    description: description ?? undefined,
    logo: stripCacheBuster(logo),
    url: absoluteUrl(path),
    sameAs: website ? [website] : undefined,
  };
}

export function imageObjectSchema({
  url,
  caption,
  description,
  representativeOfPage,
}: {
  url: string;
  caption: string | null;
  description?: string | null;
  representativeOfPage?: boolean;
}) {
  const clean = stripCacheBuster(url)!;
  return {
    "@type": "ImageObject",
    contentUrl: clean,
    url: clean,
    caption: caption ?? undefined,
    description: description ?? undefined,
    representativeOfPage: representativeOfPage || undefined,
  };
}
```

**Note:** `JSON.stringify` drops `undefined` properties, so the helpers can freely emit `undefined` for absent fields — no conditional spreading needed.

**PostgREST duplicate-embed note:** the photo-level `photo_items` embed is **aliased** `tagged:photo_items(...)`. `ITEM_SUMMARY_SELECT` itself contains an `items`-level `photo_items(...)` (for the item thumbnail).
```ts
export async function fetchPhotoById(id: string): Promise<PhotoDetail | null> {
  const { data, error } = await supabase
    .from("photos")
    .select(
      `id,image_url,alt_text,title,description,created_at,updated_at,
       project_photos(order,projects(id,slug,title,year,location,status,project_categories(categories(name)))),
       tagged:photo_items(is_main,order,items(${ITEM_SUMMARY_SELECT}))`
    )
    .eq("id", id)
    .maybeSingle();
  if (error) throw error;
  if (!data) return null;

  // Public iff the photo belongs to a PUBLISHED project (mirrors fetchPhotos).
  // Item-gallery-only photos get no indexable detail page.
  const publishedLink = ((data as Raw).project_photos ?? []).find(
    (pp: Raw) => pp.projects?.status === "published"
  );
  if (!publishedLink) return null;

  const feed = normalizePhotoFeed({ ...data, project_photos: [publishedLink] });
  const items: ItemSummary[] = [...((data as Raw).tagged ?? [])]
    .sort((a: Raw, b: Raw) => Number(!!b.is_main) - Number(!!a.is_main) || ord(a.order) - ord(b.order))
    .map((r: Raw) => r.items)
    .filter(Boolean)
    .map(normalizeItemSummary);

  return { ...feed, items };
}
```

```tsx
function photoTitle(p: { title: string | null; alt: string | null; projectTitle: string | null }) {

export async function generateMetadata({ params }: Props): Promise<Metadata> {
```

```ts
async function jsonLdTypes(page): Promise<string[]> {
  const scripts = page.locator('script[type="application/ld+json"]');
  const types: string[] = [];
  for (let i = 0; i < (await scripts.count()); i++) {
    const parsed = JSON.parse((await scripts.nth(i).textContent()) ?? "{}");
    const nodes = parsed["@graph"] ?? [parsed];
    for (const n of nodes) if (n && n["@type"]) types.push(n["@type"]);
  }
  return types;
}
```
