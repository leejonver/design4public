## ⚠️ 발견된 어려움과 해결 방안

### 1. Next.js App Router Route Handler 테스트
**문제:**
```
TypeError: Cannot directly test Route Handlers in Jest
```

**원인:**
- Next.js 14 App Router는 Jest에서 직접 테스트 불가
- Route Handler의 런타임 환경이 Edge Runtime 또는 Node.js Runtime에 의존

**해결 방안:**
1. ✅ **E2E 테스트로 대체** (권장) - Playwright로 실제 API 호출 테스트
2. 비즈니스 로직을 별도 함수로 분리하여 단위 테스트
3. Next.js의 실험적 `next/test` 유틸리티 사용

**적용:**
- E2E 테스트에서 로그인 API를 간접적으로 검증
- 향후 API Routes는 모두 E2E로 테스트 예정

---

### 2. Supabase RPC 함수 모킹
**문제:**
```
TypeError: supabaseAdmin.rpc is not a function
```

**원인:**
- Supabase 클라이언트의 RPC 메서드를 완전히 모킹하기 어려움
- 복잡한 체이닝 메서드 구조

**해결 방안:**
1. ✅ **실제 Supabase 테스트 프로젝트 사용** (권장)
2. Supabase 클라이언트 전체를 커스텀 모킹
3. E2E 테스트로 실제 데이터베이스 작업 검증

**향후 계획:**
- Supabase 테스트 프로젝트 생성
- 테스트 전후 데이터베이스 초기화 스크립트 작성
- CI/CD에 통합

---

### 3. Ant Design 컴포넌트 모킹
**문제:**
```
Error: Element type is invalid ... UserOutlined
```

**원인:**
- Ant Design의 아이콘과 복잡한 컴포넌트를 모킹하기 어려움
- Jest + jsdom 환경에서 제대로 렌더링 안 됨

**해결 방안:**
1. ✅ **E2E 테스트로 대체** (권장) - 실제 브라우저에서 테스트
2. Ant Design을 완전히 모킹하는 대신 스냅샷 테스트 사용
3. 컴포넌트를 더 작은 단위로 분리

**적용:**
- Sidebar 테스트를 E2E로 작성
- 네비게이션 동작을 실제 브라우저에서 검증

## 💡 얻은 인사이트

### 1. 테스트 전략
- **단위 테스트:** 순수 함수와 유틸리티에 집중
- **통합 테스트:** 비즈니스 로직과 상태 관리
- **E2E 테스트:** UI, API Routes, 사용자 플로우

### 2. Next.js App Router 테스트
- Route Handler는 Jest로 직접 테스트하기 어려움
- E2E 테스트가 더 효율적이고 신뢰할 수 있음
- 로직을 별도 함수로 분리하면 단위 테스트 가능

### 3. Supabase 테스트
- 복잡한 모킹보다 실제 테스트 DB 사용이 좋음
- RPC 함수는 특히 모킹이 어려움
- E2E 테스트가 더 실용적

### 4. Ant Design 테스트
- 복잡한 UI 라이브러리는 모킹이 어려움
- E2E 테스트가 더 안정적
- 스냅샷 테스트는 제한적으로 유용
