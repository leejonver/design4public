# Milestone 3: Local Supabase E2E Harness + CI — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up a fully local Supabase (Docker) E2E stack — baseline schema migration authored from the production backup dump, deterministic seed data, and a Playwright suite (site + admin + integration) plus GitHub Actions CI — that exercises the merged app end-to-end with **zero production DB access**.

**Architecture:** `supabase` CLI project (`supabase/config.toml` + `migrations/` + `seed.sql`) boots a local Postgres+Auth+Storage stack on `127.0.0.1`. Playwright's `webServer` starts `next dev` with env pointed at that local stack; a Playwright *setup project* asserts the target host is loopback (hard-fail otherwise), provisions a master + a content-manager test user against the LOCAL auth admin API, and saves `storageState` for reuse. Three suites run: `site/` (parallel-safe, read-only public pages), `admin/` (serial, authenticated CRUD), `integration/` (serial, the original "admin edit → site reflects" revalidation regression). CI runs lint + tsc + vitest + `supabase start` + Playwright (chromium) on every PR.

**Tech Stack:** Next.js 14.2.32 (App Router), `@supabase/ssr` + `@supabase/supabase-js`, Supabase CLI 2.98.2 (Docker), Playwright 1.55, Vitest 3, GitHub Actions.

## Global Constraints

- **Production DB is untouchable.** No production URL or key may appear in any test config, migration, seed, workflow, or `.env.test`. The baseline migration is authored **from the backup dump** (`/Users/jaehwanlee/development/d4p/backups/schema_20260703_051836.sql`), never `supabase db pull`. `global.setup.ts` MUST assert the Supabase URL host is `127.0.0.1` or `localhost` and **throw** otherwise (spec §9, §12).
- **Local Supabase endpoints (fixed CLI defaults):** API `http://127.0.0.1:54321`, DB `postgresql://postgres:postgres@127.0.0.1:54322/postgres`. The local anon/service-role keys are the well-known Supabase CLI demo JWTs (stable while `auth.jwt_secret` is left at the CLI default) — they are public dev keys, safe to commit in `.env.test`.
- **App env var names (unchanged):** `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`. Provisioning also uses `E2E_MASTER_EMAIL`/`E2E_MASTER_PASSWORD` and `E2E_CM_EMAIL`/`E2E_CM_PASSWORD`.
- **All admin URLs are under `/admin/*`** (App Router, no basePath). The public site uses the `(site)` route group so its URLs are at root (`/`, `/projects`, …). The two existing specs in `tests/e2e/admin/` reference the wrong pre-merge paths (`/login`, `/projects`) and must be corrected to `/admin/login`, `/admin/projects`.
- **Migrations are additive-only** (spec §12). This milestone adds a NEW baseline migration; it does not modify or apply production. Legacy `migrations/00x_*.sql` (numbered SQL) are NOT the supabase-CLI migration set — they stay as-is; the CLI migrations live in `supabase/migrations/`.
- **Playwright:** admin suite serial (shared `storageState`), site suite parallel-safe. Chromium only. Retries 2 in CI.
- **Branch:** all work on `unify/m3-e2e-harness`, branched from `unify/m2-revalidation`. Frequent commits per task.
- **Deliberate baseline omissions:** the `backup_020`/`backup_022` schemas and the unused `public.project_image_type` composite type in the dump are legacy backup artifacts — they are NOT reproduced in the baseline. Everything the app touches (tables, enums, functions, constraints, indexes, RLS, triggers) IS reproduced.

create or replace function public.handle_new_user() returns trigger
  language plpgsql security definer as $$
begin
  insert into public.profiles (id, email, role, status)
  values (new.id, new.email, 'content_manager', 'pending')
  on conflict (id) do nothing;
  return new;
end;
$$;

  language plpgsql as $$
begin
  new.updated_at = timezone('utc'::text, now());
  return new;

```ts
import { test as setup, expect } from '@playwright/test'
import { createClient } from '@supabase/supabase-js'
import path from 'node:path'

const URL = process.env.NEXT_PUBLIC_SUPABASE_URL!
const SERVICE_KEY = process.env.SUPABASE_SERVICE_ROLE_KEY!

// SAFETY NET (spec §9/§12): the E2E harness must never touch a remote DB.
// Hard-fail at import time if the Supabase URL is not loopback.
const host = new URL(URL).hostname
if (host !== '127.0.0.1' && host !== 'localhost') {
  throw new Error(
    `E2E refuses to run against non-local Supabase host "${host}". ` +
      `Expected 127.0.0.1/localhost. Check .env.test.`,
  )
}

const admin = createClient(URL, SERVICE_KEY, {
  auth: { autoRefreshToken: false, persistSession: false },
})

async function provision(email: string, password: string, role: 'master' | 'content_manager') {
  // Create the auth user (email pre-confirmed). Idempotent: on "already
  // registered" we look the user up and reset the known password.
  const created = await admin.auth.admin.createUser({ email, password, email_confirm: true })
  let userId = created.data.user?.id
  if (!userId) {
    const { data } = await admin.auth.admin.listUsers({ perPage: 200 })
    const found = data.users.find((u) => u.email === email)
    if (!found) throw new Error(`could not create or find user ${email}`)
    userId = found.id
    await admin.auth.admin.updateUserById(userId, { password, email_confirm: true })
  }
  // The on_auth_user_created trigger inserted a pending content_manager profile;
  // upsert it to the role/approval this test user needs.
  const { error } = await admin
    .from('profiles')
    .upsert({ id: userId, email, name: `E2E ${role}`, role, status: 'approved' })
  if (error) throw error
  return userId
}

setup('provision users and save auth state', async ({ browser }) => {
  await provision(process.env.E2E_MASTER_EMAIL!, process.env.E2E_MASTER_PASSWORD!, 'master')
  await provision(process.env.E2E_CM_EMAIL!, process.env.E2E_CM_PASSWORD!, 'content_manager')

  // Log in through the real UI (client-side @supabase/ssr sets the session
  // cookies) and persist storageState for each role.
  for (const [email, password, file] of [
    [process.env.E2E_MASTER_EMAIL!, process.env.E2E_MASTER_PASSWORD!, 'tests/e2e/.auth/master.json'],
    [process.env.E2E_CM_EMAIL!, process.env.E2E_CM_PASSWORD!, 'tests/e2e/.auth/manager.json'],
  ] as const) {
    const context = await browser.newContext()
    const page = await context.newPage()
    await page.goto('http://127.0.0.1:3000/admin/login')
    await page.getByPlaceholder('이메일 주소').fill(email)
    await page.getByPlaceholder('비밀번호').fill(password)
    await page.getByRole('button', { name: /로그인/ }).click()
    await page.waitForURL(/\/admin\/projects/, { timeout: 15000 })
    await context.storageState({ path: path.resolve(__dirname, '..', '..', file) })
    await context.close()
  }
})
```
